# 引き継ぎメモ

このファイルは README や ルール文書とは別に、実作業で躓いた点や 今後の注意点を残すためのメモです。  
別のAIや 後日の自分が見ても、今の状態と判断基準が追えることを目的にします。

## 使い方

- 新しく躓いたことがあれば `躓きメモ` に追記する
- 直したことは `解決済み` に移す
- 後でやることは `未完了` に残す
- 共通対応で済むか 個別確認が必要かを明記する

## 現在の状態

- GitHub Pages の静的サイト
- 全機種分の記事データJSONは作成済み
- 北斗ページは現時点の基準ページとして使っている
- ただし 北斗ページ自体も最終版ではなく 後で追加項目とブラッシュアップが入る
- ただし 北斗ページと比べると 情報量と精度はまだ不足
- 今は ページ数を増やす段階から 内容の精度と厚みを上げる段階へ移行済み

## ルールの要点

- このプロジェクトでは `CODEX_RULES.md` `HANDOFF.md` `README.md` の3ファイルを通称「ルール」と呼ぶ
- 会話中に「ルールを確認」「ルールに記載」などの指示があれば 上記3ファイルを指すものとして扱う
- ルール文書は `CODEX_RULES.md` を正本として扱う
- 新スレッドや初回着手時は まず全ファイル一覧を確認して 実際の構造を把握する
- ただし 全ファイルを最初から全文確認する必要はない
- 全文確認は ルール 主要ファイル 変更対象 関連ファイルの順で優先する
- ルールに書かれた構造と 実ファイル構成に差分がないかを見る
- スクショ確認用の一時実装は 公開用仕様に混ぜず 必要な便だけ差し込み 確認後に外す
- 変更前に必ず現在ファイルを確認する
- いきなり実装せず 先に修正方針を出す
- ユーザーのOK後に実装する
- 軽微な修正でも同じ手順を守る
- 関連する修正は なるべくまとめて入れる
- 公開確認は クリックできるURLで案内する
- スクショ確認中心でも進められるようにする
- 複数URLを出すときは 必ず `1行に1URL` の貼り付け用テキストでも出す
- 修正や変更を加えた後は 確認用URLをコピーしやすい形で貼り付ける
- 確認用URLは 原則としてコードブロック内にURLだけを並べて 1クリックでコピーできる形で出す
- 複数URLを貼るときは 各URLの末尾で1回だけ改行し 余分な空行は入れない
- ツールに貼るURLは `スクショ用URL` として明示して出す
- 実装後に確認スクショが未回収の便では 次の修正へ進む前に 未回収であることを明示する
- 確認スクショ待ちの段階で 次の作業へ進みそうなら 先に止めて案内する
- スクショ確認のためだけに必要な URLパラメータや状態指定は 常設機能にしない
- 上の確認用実装は 必要な便だけ一時的に入れ 確認が終わったら抜く
- アップ先ディレクトリは 普通の文章で出す コードブロックにしない
- 作業の区切りごとに `CODEX_RULES.md` `README.md` `HANDOFF.md` の更新要否を確認する
- 実装後は アップ先ディレクトリとコミットサマリーも毎回まとめる
- GitHub Desktopでコミットするためのコミットサマリー文を そのままコピーできる形で出す
- 修正や変更を加えたら そのターンで何をしたかを `CODEX_RULES.md` `README.md` `HANDOFF.md` のどれかへ必ず記録する
- まだコミットやアップの便を切らない段階では コミットサマリーを先に出さない
- コミットサマリーを今出すべきか 保留すべきかは 毎回自分で判断する
- コミット可能な変更がある便では コミットするか保留するかの判断を返答前に必ず確認する
- コミットする便では コミットサマリー文の出し忘れをしない
- ユーザーから 差分が見えない と言われた場合は 確認用コメントや確認用1行を入れて差分の見え方を確認してよい
- 上の確認用コメントや確認用1行は 確認後に残すか外すかを都度判断する
- 3ファイルの更新は 基本は作業の区切りでまとめて行う
- ただし 後から見返すと重要な判断や運用変更は 区切り前でも必要時点で追記する
- コミットサマリーは 実際にコミット対象の変更がある時だけ出す
- 確認だけで新しい変更がないターンでは コミットサマリーを出さない
- 3ファイルの更新は 基本的に大きな修正と同じ便でまとめてアップする
- ただし 後から抜けると困る重要な運用変更や判断基準は 単体更新と単体アップを許容する
- 運用ルールだと判断した指示は 確認待ちせず必要なファイルへ自動で追記する

## 躓きメモ

### 記事ページの文体

- 北斗ページを参考にするはずが 一時期 テンプレ感の強いAIっぽい文章になった
- 数値の羅列や `caution / good / excellent` 直出しは公開文として弱かった
- 解決方針:
  - 北斗ページの章立てに寄せる
  - ただし内容まで北斗に揃えず 機種固有の説明は足す
  - 詳細本文では 必要な `、` `。` を入れる

### 早見表と基本情報

- 北斗以外は `狙い目早見表` と `基本情報` に似た情報が重複しやすい
- 現状は ページ成立を優先しているため 役割分担はまだ甘い
- 後で 北斗ページのように `要点` と `別情報` に分け直す必要あり

### 設定示唆まとめの表幅

- 表ごとに wide を付けるだけだと 同じページ内で上下の表がでこぼこに見える
- 現在は 同じページ内なら右列幅を統一するように修正済み
- これで かぐや様のような上下の表幅不一致は改善済み

### 設定示唆まとめの右列バッジ

- 短文でも 1文字だけ折り返して見た目が悪くなることがある
- 全部を改行禁止にすると 狭い画面で逆に崩れやすい
- 方針:
  - 共通CSSで `1行優先`
  - ただし長文だけは 機種ごとに個別確認して必要なら調整

### 設定示唆まとめの色

- 現状は 金系3段階 + 緑1段階
- 意味としては `緑 > 濃い金 > 中金 > 薄金`
- ただし 見た瞬間に 緑が最上位と伝わりにくい
- 後で 配色の改善候補あり
- それでも分かりにくければ 色つき凡例の短文を上に足す案あり

### SF5 の扱い

- SF5 はゲーム数チェッカーではなく 設定狙い向けページとして扱う
- `ハイエナ` という語は公開文では使わない
- 採用済み表現:
  - `ゲーム数狙いには向きません`
  - `設定狙い向けの機種です`

### スクショ確認運用

- ユーザーは全ページを細かく見るのではなく 気になるページだけスクショを送る運用
- ユーザーは 複数URLをそのまま投げてスクショ生成できるため URLは改行区切りのまま渡す
- 機種名つきフルページスクショがあれば URLなしでも判断可能
- 現在は ユーザー側に複数URLを一括撮影できるスクショ取得ツールあり
- 画像下部にURLも焼き込まれるため 機種名とURLの両方で判断できる
- 今後 確認用URLを出すときは クリックできる形に加えて `1行に1URL` の貼り付け用テキストも必ず出す
- 画像だけで判断できる範囲:
  - 表示崩れ
  - 文字詰まり
  - テーブル幅不足
  - 文言違和感
  - 余白や改行の不自然さ

### GitHub Pages の反映ラグ

- GitHub 側のチェック完了と 公開反映には時間差がある
- ローカルでは直っていても 公開ページでは少し遅れて見えることがある
- 反映確認時は そのラグを前提に案内する

### service worker とキャッシュ

- service-worker.js が入っているため 公開確認時に古いキャッシュを見やすい
- 確認用途としては `Ctrl + F5` がいちばん確実
- service worker の改善は可能だが `100%常に最新` を保証するものではない
- 確認作業優先なら 今はキャッシュ改善より ページ修正を優先する
