
このプロジェクトのルール:

■ 基本
・いきなりファイルを書き換えない
・必ず先に修正方針を提示する
・変更するファイル名を明記する
・差分は最小限にする
・不要なファイルは追加しない
・削除が必要な場合は必ず事前確認する
・既存構造を壊さない
・GitHub Pagesの静的サイトとして正常動作を最優先とする
・回答は日本語で行う

■ 作業手順
・まず変更せずに、修正方針と変更ファイル名だけ出す
・新スレッドや初回着手時は まず全ファイル一覧を確認して 実際の構造を把握する
・ただし 全ファイルを最初から全文確認する必要はない
・全文確認は ルール 主要ファイル 変更対象 関連ファイルの順で優先する
・ルールに書かれた構造と 実ファイル構成に差分がないかを見る
・ユーザーのOKが出てから実装する
・実装後は以下を必ずまとめる
  - 変更内容
  - 影響範囲
  - 確認ポイント
  - アップ先ディレクトリ
  - コミットサマリー（コミット対象がある場合のみ）

■ 回答ルール
・聞かれていない改善提案は出さない
・推測ではなく実際の記述ベースで答える
・素人にもわかりやすい説明をする
・複数URLを出すときは必ず1行に1URLで改行して出す
・修正や変更を加えた後は 確認用URLをコピーしやすい形で貼り付ける
・確認用URLは 原則としてコードブロック内にURLだけを並べて 1クリックでコピーできる形で出す
・複数URLを貼るときは 各URLの末尾で1回だけ改行し 余分な空行は入れない
・ツールに貼るURLは `スクショ用URL` として明示して出す
・アップ先ディレクトリは普通の文章で出す（コードブロックにしない）
・コミットサマリーは実際にコミット対象の変更があるときだけ出す
・まだコミットやアップの便を切らない段階では コミットサマリーを先に出さない
・コミットサマリーを今出すべきか 保留すべきかは 毎回自分で判断する
・確認だけで新しい変更がない場合はコミットサマリーを出さない
・運用ルールだと判断した指示は 確認待ちせず必要なファイルへ自動で追記する
・3ファイル更新は基本的に大きな修正と同じ便でまとめてアップする
・ただし 後から抜けると困る重要な運用変更や判断基準は 単体でも先に更新してよい

その他構造やルール

- このプロジェクトでは `CODEX_RULES.md` `HANDOFF.md` `README.md` の3ファイルを通称「ルール」と呼ぶ
- 会話中に「ルールを確認」「ルールに記載」などの指示があれば 上記3ファイルを指すものとして扱う

## 作業ルール（必ず守ること）

- ファイル変更前に必ず現在のファイルを確認してから作業する
- 初回は 全ファイル一覧を確認してから作業に入る
- ただし 全ファイル全文確認ではなく 一覧確認と主要ファイル確認を優先する
- 変更は差分のみ。新規作成で上書きしない
- 大きな変更前はプレビューを見せてから進める
- アップ先ディレクトリとコミットサマリー文をセットで記載する
- コミットサマリーはコードブロック（```）で囲んで1クリックでコピーできるようにする
- GitHub Desktopでコミットするためのコミットサマリー文を そのままコピーできる形で出す
- まだコミットしない変更を記録しただけの段階では コミットサマリーを出さない
- アップ先ディレクトリのファイルは名前順で並べる
- 確認だけで新しい変更がない場合は コミットサマリーを出さない
- **ファイル名ではなくページ名で話す**（例：`machine.html` → 「記事ページ」）
- わからないことはわからないと言う
- 辛口でアドバイスする
- 常に敬語で話す
- ユーザーはコードの素人。ユーザーがやることは動作確認のみ
- 確認してほしいことは「どのページのどこを見てほしいか」を具体的に伝える

## CSSの重要ポイント（practical.css）

### settei-table（設定示唆テーブル）
```css
.settei-table { width: 100%; table-layout: fixed; }
.settei-table th:last-child { width: 90px; }
.settei-badge { display: block; width: 100%; box-sizing: border-box; white-space: nowrap; overflow: hidden; }
```
- 記事ページにインラインstyleを絶対に書かない（過去に18%固定が埋め込まれて長期間ハマった）
- 右列90pxは「設定4以上期待大」（9文字・font-size:10px）がギリギリ収まるサイズ

### 横向き検知
```javascript
function checkOrientation(){
  const isLandscape = window.innerWidth > window.innerHeight;
  document.body.classList.toggle('is-landscape', isLandscape);
}
```
- .is-landscape クラスをbodyに付与してCSSで切り替え

### summary-grid（狙い目早見表）
```css
.summary-grid { width: 100%; border-collapse: collapse; border: 1px solid var(--line2); }
```
- `<table>`タグを使用（divグリッドではない）

### 優先度2：交換率4パターン対応（大規模改修）
**背景**：知り合いのプロスロプレイヤーから「等価・非等価の切替がないとハイエナユーザーには使えない」との指摘あり。

**設計方針**：
- 交換率は「等価（5.6枚）」「5.5枚」「5.0枚」「4.5枚」の4択
- 初回だけ選択してlocalStorageに保存・以後自動適用
- machines.jsonの各モードの数値を4交換率分持つ構造に変更
- 全18機種 × 4交換率 × モード数分の数値調査が必要

**実装手順**（この順番で進めること）：
1. 全機種の調査未完了分を終わらせる
2. 全機種の数値を4交換率×モード別で一覧表にまとめる
3. machines.jsonのJSON構造を設計・一括反映する
4. 記事ページとチェッカーページに交換率セレクターUIを追加する

### 進め方の補足

- 軽微な修正でも 変更前に必ず現在ファイルを確認する

- 3ファイルへ追記しただけで まだアップの便を切らない場合は コミットサマリーを出さず 記録だけ残す
- 3ファイルの更新は基本的に作業の区切りでまとめて行う

- 3ファイルを更新した場合も 基本は大きな修正と同じ便でまとめてアップする

## 文書の役割分担

- `README.md`
  - サイト概要 構成 新機種追加の手順だけを見る
- `CODEX_RULES.md`
  - 作業前の手順 回答ルール 守るべき運用を見る
- `HANDOFF.md`
  - どこで躓いたか どう直したか 未完了事項 今後100機種以上へ増やす時の注意点を見る

## 新スレ移行ルール

- 新しいスレッドへ移る前には `CODEX_RULES.md` `README.md` `HANDOFF.md` に再開地点と完了範囲を書く
- 新スレッド開始時は まず `CODEX_RULES.md` と `HANDOFF.md` を確認してから作業を再開する
